# 요구사항 정의 (Requirements)

프로젝트의 요구사항 정의 및 분석을 담당한다. 모호하고 비정형적인 사용자 요건을 정형화된 기술 명세서(`docs/spec.md`, 단일 SSOT)로 도출·변환하고, 요구사항의 변경 이력을 통제한다.

---

## 담당 작업 범위

```text
✅ 자율 실행 가능
  - 신규 요구사항 템플릿 복사 및 초기 세팅
  - 사용자 질문 답변을 통한 docs/spec.md 요구사항 파트 초안 작성 및 상세화
  - 기능 요구사항의 계층화(F-1.1, F-1.2 등) 및 입력/출력 인터페이스 규정

⚠️ 사용자 승인 필요
  - docs/spec.md의 변경 및 갱신 적용
  - 이미 승인 완료된 기능 항목의 삭제 또는 구현 범위 변경
  - API 사양 중 필수 데이터 필드의 규격 변경
```

---

## 부트스트랩 및 문서화 라이프사이클
1. **디렉토리 검사**: 프로젝트 첫 세션 진입 시 루트에 `docs/` 폴더가 있는지 검사하고, 없으면 생성한다.
2. **템플릿 이식**: [`templates/SPEC_TEMPLATE.md`](../../reference/templates/SPEC_TEMPLATE.md)를 읽어 `docs/spec.md` 경로에 복사한다.
3. **순차 빌드업**: 복사된 `docs/spec.md`를 아래 Q&A 가이드라인으로 채워 나간다.

## 상세 요건 도출 가이드라인 (자가 문답)
문서를 채우기 전 아래 5가지 질문에 스스로 혹은 사용자에게 물어 명확한 답을 확보한다.
1. **비즈니스 목적**: 이 시스템이 해결하려는 핵심 고통(Pain Point)은 무엇인가?
2. **데이터 원천**: 입력 데이터는 어디서 제공되는가? (사용자 직접 입력 / 외부 REST API / 로컬 DB)
3. **성능 타임아웃 한계**: 해당 API 응답은 최대 몇 초 이내에 반환되어야 하는가? (기본값: 500ms)
4. **보안 규정**: 어떤 수준의 권한 제어가 필요한가? (비로그인 허용 / JWT 일반 회원 / 관리자 전용)
5. **예외 동작**: 입력값 검증 실패 시 어떤 오류 메시지와 HTTP 코드를 보여줄 것인가?

## 요구사항 상충(Conflict) 및 비대화형 배치 환경 예외 분기 정책 (Strict Fail-Fast)
* **비대화형 배치(CI/CD)**: 사용자 피드백이 불가한 자동화 파이프라인에서 요구사항 모순이 감지되면, 임시로 문서화하고 통과시키는 우회를 절대 금지한다. 모순 감지 즉시 실패(Exit 1)를 트리거하고 상세 결함 리포트를 생성해 즉시 예외를 보고한다.
* **실패 보고 템플릿**:
```text
[요구사항 상충에 따른 빌드 실패 경고]
제시해주신 요구사항 간에 기술적 모순이 존재하여 빌드 게이트를 중단합니다. (Exit 1)
1. 성능 요건: 응답시간 500ms 이하
2. 동기 데이터 처리: 10만 건 엑셀 동기 업로드

[피봇팅 제안 옵션 (수동 승인 대기)]
- 옵션 A: 비동기 처리 구조로 전환 (업로드 즉시 202 Accepted 반환 후 백그라운드 처리, 화면에서 상태 폴링)
- 옵션 B: 성능 기준치 완화 (동기 업로드 완료 시까지 최대 15초 대기 허용)
인간 개발자의 수동 해결 및 명시적 spec.md 승인이 완료될 때까지 배포가 동결됩니다.
```

## 환경 인프라별 비기능 요건 다변화 정책
성능 임계값을 일률적으로 지정하지 않고 테스트 구동 환경에 맞춰 이원화한다.
* **로컬 개발 환경**: GPU 부재·로컬 CPU 환경을 고려해 LLM API 응답 타임아웃 한계를 3,000ms로 버퍼 설정.
* **운영 배포 환경**: 고성능 다중 컨테이너 인프라를 고려해 500ms 이내 응답 강제.

## 기능 명세의 정밀 기록 스펙
모호한 자연어를 금지하고 입력·출력을 엄격히 규정한다.
* *안티패턴*: "사용자가 데이터를 조회할 수 있어야 함."
* *모범패턴*:
  * **기능 ID**: F-2-1 (데이터 상세 조회)
  * **입력 매개변수**: `user_id` (UUID, 필수), `query_date` (YYYY-MM-DD, 선택)
  * **비즈니스 규칙**: `user_id`가 존재하지 않으면 `404 Not Found`, `query_date`가 미래 날짜면 `400 Bad Request`.
  * **출력 구조**: `{"status": "success", "data": {"records": []}}`

## spec.md 통합 및 변경 통제
* `docs/spec.md` 작성 완료 시 핵심 비즈니스 목적과 마일스톤을 요약 섹션으로 유지한다 (별도 파일 동기화 불필요).
* **스펙 락(Spec Lock)**: 한 번 합의되어 `spec.md`에 적용된 요구사항은 단독 수정을 금지하며, 반드시 사용자의 "승인(Proceed)" 절차를 거친 이후에만 변경을 수용한다.

## 금지 사항
* 🚫 "성능은 빨라야 한다"처럼 측정 불가능한 비기능 요구사항 정의 금지 (반드시 수치로 기록).
* 🚫 사용자 승인 없이 `spec.md`에 정의된 기존 기능 스펙을 강등하거나 삭제하는 행위 금지.
* 🚫 입력 파라미터의 데이터 타입/필수 여부를 누락하고 API 명세를 작성하는 행위 금지.
