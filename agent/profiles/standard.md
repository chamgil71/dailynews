# Standard Profile (일반 프로젝트 모음)

이 프로필은 이 워크스페이스의 정식 프로젝트들이 따르는 표준 에이전틱 코딩 지침입니다. (예: `invest-platform`, `myassets`, `mymap` 등)

---

## 1. 거버넌스 및 문서화 규칙
* **단일 진실 공급원(SSOT) 준수**: `AGENTS.md`를 기점으로 작업 전 요구사항 및 설계를 명문화합니다.
* **실시간 문서 갱신**: 개발 전 [spec.md](./docs/spec.md)에 기능을 명세하고, 개발 과정의 세부 이행 로그를 [worklog.md](./docs/worklog.md)에 상세히 기록합니다.

---

## 2. 아키텍처 원칙 (4-Layer 및 백엔드 도입 4대 제한 기준)
* **표준 구조**: `frontend/`, `backend/`, `docs/`, `storage/` 계층 분리를 원칙으로 합니다.
* **백엔드(FastAPI 등) 도입 4대 제약**: 단순 연산이나 고정 정적 뷰를 처리할 때는 백엔드 서버를 절대 임의 신설하지 않으며, 아래 4가지 조건 중 **하나 이상 만족할 때만** 백엔드 계층을 신설합니다.
  1. **중앙 데이터베이스 영속 계층 필요**: 다수의 동시 쓰기/조회 및 정교한 SQLite/DB 연동이 핵심일 때.
  2. **보안/인증 처리**: OAuth, JWT 등 외부 API 토큰 유출 방지가 필요할 때.
  3. **외부 API 프록시**: 브라우저 CORS 제약을 우회해야 하거나 백엔드 전용 API 연동이 필요할 때.
  4. **무거운 서버 연산**: 대규모 데이터 수집/전처리, ML/LLM 오케스트레이션 등 클라이언트 부담이 큰 로직일 때.

---

## 3. 개발 스타일 및 품질
* **파일 분리 컨벤션**: 단일 파일의 소스 코드가 **200라인을 초과**하는 경우, 도메인이나 유틸로 로직을 정밀 분리 및 격리합니다.
* **타입 힌팅 및 라이브러리**: Python의 경우 `pathlib.Path` 사용을 강제하고, Pydantic v2 validation(v1 deprecation 패턴 금지)을 철저히 준수합니다.
* **자동화 검증**: 핵심 기능에 대한 pytest 유닛 테스트를 의무 작성하여 정상 통과(passed) 상태에서 턴을 완료합니다.
