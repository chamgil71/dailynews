# 에이전트 프로그래밍 개발 스타일 가이드 (development_style_guide.md)

본 문서는 프로젝트 소스코드 구현 시 에이전트가 준수해야 할 언어별, 프레임워크별 실무 코딩 스타일 및 코드 구조 표준을 정의합니다.

---

## 📌 최상위 공통 원칙 (200라인 제한 및 경량 문서화)
*   **단일 소스 파일 200라인 제한**: 가독성과 모듈러 구조 확보를 위해, 단일 소스 코드 파일(Python `.py`, React `.js/.jsx/.ts/.tsx` 등)의 총 길이는 **200라인을 초과할 수 없습니다.** 200라인을 초과하기 직전, 논리적 도메인이나 공통 유틸리티 모듈로 즉각 분리 및 리팩토링해야 합니다.
*   **경량 문서화 동기화**: 모든 설계와 구현은 무거운 사양서 대신 `docs/spec.md` (기능 명세)와 `docs/todo.md` (혹은 `worklog.md` 작업 기록)를 중심으로 기동하며, 구현 완료 즉시 실시간으로 문서를 업데이트하여 현행화합니다.

---

## 1. Python 개발 표준 규칙

모든 Python 코드는 다음 기준을 엄격히 준수합니다.

### 1.1 스타일 및 네이밍 (PEP8 표준)
*   **들여쓰기**: 4 spaces (Tab 사용 금지)
*   **최대 라인 길이**: 100자 이하
*   **명명 규칙**:
    *   변수 및 함수/메서드: `snake_case` 사용
    *   클래스: `PascalCase` 사용
    *   상수: `UPPER_SNAKE_CASE` 사용
*   **타입 힌트 (`typing`)**: 모든 함수의 입력 매개변수와 반환값에 타입 힌트를 의무적으로 명시합니다.
    *   *예시*: `def find_user_by_id(user_id: int) -> Optional[User]:`
*   **Docstring**: 모든 public 함수, 클래스, 모듈에는 기능의 목적과 입출력을 설명하는 삼중 따옴표 docstring을 작성합니다.

### 1.2 경로 연산 및 Pydantic v2 표준 규칙
*   **pathlib 사용 강제**: 모든 디렉토리 및 파일 경로 다루기 시, 문자열 포맷팅이나 문자열 덧셈 연산을 일체 금지하며 반드시 `pathlib.Path` 객체와 `/` 연산자를 사용합니다.
    *   *예시*: `target_path = Path("C:/coding") / "storage" / "data"`
*   **Pydantic v2 엄격 준수**: 데이터 직렬화 및 검증용 Pydantic 모델 작성 시 v2 문법을 필수적으로 사용하며, v1의 deprecated 패턴(예: `@validator`, `@root_validator` 등)의 사용을 금지합니다. (반드시 `@field_validator`, `@model_validator` 등을 적용할 것)

### 1.3 에러 처리 및 로깅
*   **명시적 예외 처리**: `try/except` 블록 사용 시 빈 `except:` 또는 `except Exception: pass` 형태로 예외를 무시(Swallow)하는 패턴을 금지합니다. 로깅 후 상위로 재전파(`raise`)하거나 명확한 fallback 값을 반환해야 합니다.
*   **로깅**: `print()` 사용을 전면 금지하며, `logging` 라이브러리를 통해 적절한 레벨(`info`, `warning`, `error`)로 로그를 기록합니다.

### 1.4 함수 설계 표준 규칙
*   단일 함수의 길이는 기계적인 줄 수 제약 대신 아래의 3단계 완충 기준 및 다차원 복잡도를 종합적으로 검토하여 코드를 관리합니다.
    *   **권장 (≤ 50줄)**: 함수는 단일 책임(SRP)을 성실히 수행하며 50줄 이하의 간결한 구조를 유지하는 것을 지향합니다.
    *   **주의 (51 ~ 100줄)**: 상태 머신, 파서, 복잡한 수식 알고리즘 등 구조적으로 한 곳에 묶여 있는 것이 더 명확한 경우 예외적으로 허용합니다. 단, 단일 책임을 해치지 않는지 재검토합니다.
    *   **리팩토링 권고 (> 100줄)**: 원칙적으로 함수가 너무 비대해진 'God Function' 상태로 규정하고 즉각적인 분할 및 리팩토링 대상으로 취급합니다.
*   **다차원 복잡도 및 품질 자가 검증**:
    *   [ ] 해당 함수가 오직 단 하나의 책임만을 전담하는가 (SRP 준수 여부)
    *   [ ] 중첩 제어문(`if`, `for`, `while`)의 깊이가 3단계 이하로 제어되는가 (가독성 확보)
    *   [ ] 함수의 순환 복잡도(Cyclomatic Complexity)가 10 이하로 유지되는가
    *   [ ] 동일하거나 유사한 코드 패턴이 반복되어 나타나지 않는가 (DRY 원칙)
    *   [ ] Mocking 등을 통해 다른 컴포넌트 없이도 독립적인 유닛 테스트 작성이 용이한가

## 2. React (Frontend) 개발 표준 규칙

모든 React 프론트엔드 코드는 현대적이고 선언적인 스타일을 따릅니다.

### 2.1 React 18+ 버전 규칙
*   **함수 컴포넌트 강제**: 모든 UI 컴포넌트는 `Function Component`로만 구현합니다.
*   **Hooks 기반 상태 관리**: `useState`, `useEffect`, `useMemo`, `useCallback` 및 Custom Hook을 사용하여 로직을 제어합니다.
*   **전면 금지 패턴**:
    *   ❌ `Class Component` 작성 절대 금지
    *   ❌ `componentDidMount`, `componentWillUnmount` 등 레거시 Lifecycle API 사용 전면 차단
*   **API 통신 격리**: 컴포넌트 내부에서 직접 `fetch`나 `axios`를 호출해 비즈니스 API를 요청하는 것을 금지하며, 반드시 `services/` 레이어의 격리된 함수를 호출하여 사용합니다. (React Query/SWR 권장)

## 3. FastAPI (Backend) 개발 표준 규칙 및 도입 제약

### 3.1 소규모 프로젝트 백엔드 신설 제한 4대 기준
단순 데이터 뷰어나 수동 스크립트 도구 단계에서는 백엔드 서버를 절대 임의로 개설하지 않으며, 아래 4가지 조건 중 **최소 하나 이상 충족하는 경우에만** 백엔드 API 서버를 신설할 수 있습니다.
1. **중앙 데이터베이스 연동 및 다중 쓰기 제어**: SQLite 등 영속 계층의 트랜잭션 및 동시성 제어가 필수적일 때.
2. **보안 및 인증 처리**: OAuth2, JWT 및 민감 기밀 키(External Secret Keys) 우회 프록시가 필요할 때.
3. **외부 API 프록시 및 CORS 우회**: 브라우저 단독 통신 제약을 우회하기 위한 서버 단의 프록싱이 필요할 때.
4. **무거운 서버 연산**: ML 모델 인퍼런스, 백그라운드 데이터 수집 파이프라인 등 서버 자원을 집중 소모할 때.

### 3.2 수직 레이어 단방향 의존성
*   호출 방향은 항상 `router ──► service ──► repository ──► database` 단방향을 준수합니다.
*   ❌ `router`에서 `repository`를 직접 호출하거나 DB를 조작하는 행위 금지
*   ❌ `repository`에서 `service`를 역참조하는 순환 의존성 금지
*   FastAPI의 **Dependency Injection (`Depends`)** 패턴을 사용하여 레이어 간의 결합도를 낮춥니다.

### 3.3 입력 검증 및 비동기 처리
*   모든 API 요청과 응답 스키마는 **Pydantic Model**로 정의하여 타입 안정성과 데이터 검증을 강제합니다.
*   FastAPI의 비동기 이점을 극대화하기 위해 endpoint는 `async def` 사용을 기본으로 삼습니다.

## 4. Streamlit UI 개발 표준 규칙

*   **UI 레이어 격리**: Streamlit 코드는 순수한 UI 및 입력 렌더링용으로만 사용합니다.
*   ❌ Streamlit 코드 내에 SQL 쿼리 작성 또는 직접적인 DB 연결 금지
*   ❌ 복잡한 데이터 분석 및 비즈니스 연산 로직 포함 금지 (반드시 `services/` 레이어 호출 결과를 받아 렌더링할 것)
*   **상태 관리**: UI 상태와 세션 데이터는 `st.session_state`를 일관되게 활용하여 추적합니다.
