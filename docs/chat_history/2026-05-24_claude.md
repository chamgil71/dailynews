# AI Daily News — 프로젝트 문제점 분석 및 개선 제안

> **작성일**: 2026-05-24

## 요약 (Executive Summary)

전체 파이프라인(RSS 수집 → LLM 분석 → 사이트 빌드 → 이메일/텔레그램 발송 → Vercel 배포)은 잘 설계되어 있으며 운영 중입니다. 그러나 코드 리팩터링 과정에서 발생한 **경로 버그 1건이 구독 취소 기능을 무력화**시키고 있으며, **SSL 인증서 검증 비활성화**는 보안 취약점입니다. 그 외 중복 코드·레거시 shim 파일·미구현 기능이 누적되어 있습니다.

## 🔴 즉시 수정 필요 (Critical Bugs)

### [BUG-01] `_UNSUB_FILE` 경로 계산 오류 — 구독 취소 기능 비작동

**파일**: `core/shared/mailer.py:47`

```python
# 현재 코드 (잘못됨)
_UNSUB_FILE = Path(__file__).parent.parent / "storage" / "unsubscribed.txt"
# 결과 경로: C:\ai\dailynews\core\storage\unsubscribed.txt  ← 존재하지 않음
```

**원인**: 원래 `core/mailer.py`에 있던 코드를 `core/shared/mailer.py`로 이전할 때 `parent` 깊이를 조정하지 않음.  
`__file__`이 `core/shared/mailer.py`이므로 `.parent.parent` = `core/` 에 그치고 프로젝트 루트에 못 미침.

**영향**: `_get_unsubscribed()`가 항상 빈 set을 반환 → 구독 취소된 이메일에도 계속 발송됨.  
(Vercel의 `api/unsubscribe.py`는 정상적으로 `storage/unsubscribed.txt`를 업데이트하지만, 발송 시 이 파일을 읽지 못함)

**수정**:
```python
# 수정 코드
_UNSUB_FILE = Path(__file__).parent.parent.parent / "storage" / "unsubscribed.txt"
# 결과 경로: C:\ai\dailynews\storage\unsubscribed.txt  ✓
```

### [ISSUE-04] 이메일 발송 중 개별 수신자 실패 처리 부재

**파일**: `core/shared/mailer.py:277~308`

```python
for email in recipients:
    # ...
    smtp.sendmail(GMAIL_USER, email, msg.as_string())
    success_count += 1  # ← sendmail 예외가 나면 루프 전체가 중단됨
```

`smtp.sendmail()`이 특정 수신자에서 예외를 던지면 나머지 수신자에게 발송이 중단됩니다. `except` 블록이 루프 밖에 있어 하나의 실패가 전체 발송을 중단시킵니다.

**수정**:
```python
for email in recipients:
    try:
        # ... 렌더링 ...
        smtp.sendmail(GMAIL_USER, email, msg.as_string())
        success_count += 1
        logger.info(f"[이메일 발송] → {email}")
    except Exception as e:
        logger.error(f"[이메일 발송 실패] {email}: {e}")
        # 다음 수신자로 계속
```

### [ISSUE-06] Naver 뉴스 HTML 태그 제거 불완전

**파일**: `core/stock/collector.py:111`

```python
"title": item.get("title", "").replace("<b>","").replace("</b>",""),
```

네이버 API 응답에는 `<b>`, `</b>` 외에도 `&amp;`, `&quot;`, `&#039;` 등 HTML 엔터티가 포함될 수 있습니다.

**수정**:
```python
import html, re

def _clean_naver_title(title: str) -> str:
    title = re.sub(r'<[^>]+>', '', title)  # 모든 HTML 태그 제거
    return html.unescape(title)            # HTML 엔터티 디코딩
```

## 🟡 개선 권장 (Recommended Improvements)

### [IMPROVE-01] RSS 수집 동기 처리 → 병렬화로 성능 개선

**파일**: `core/news/collector.py:168~174`

```python
for category, meta in RSS_FEEDS.items():
    for url in meta["feeds"]:
        items = _fetch_feed(url, ...)  # 피드마다 순차 대기
        time.sleep(0.3)               # 추가 지연
```

현재 20~30개 피드를 순차적으로 수집합니다. 각 피드 타임아웃(10초)이 발생하면 전체가 지연됩니다.

**개선안**:
```python
from concurrent.futures import ThreadPoolExecutor, as_completed

def collect_news() -> dict:
    # ...
    tasks = [
        (url, category, meta["lang"], meta["label"])
        for category, meta in RSS_FEEDS.items()
        for url in meta["feeds"]
    ]
    with ThreadPoolExecutor(max_workers=8) as pool:
        futures = {pool.submit(_fetch_feed, url, cat, lang, label, seen_urls): url
                   for url, cat, lang, label in tasks}
        for fut in as_completed(futures):
            all_news.extend(fut.result())
```

`seen_urls`의 thread-safety를 위해 `threading.Lock()` 추가 필요.

## 전체 아키텍처 평가

**잘 설계된 부분**:
- 피드별 장애 격리 (`_fetch_feed` 예외 시 빈 리스트 반환)
